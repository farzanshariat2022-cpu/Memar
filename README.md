# معمار (The Architect) — فازهای ۱ تا ۳ + رفع مشکل بیلد Codemagic

این نسخه شامل: احراز هویت، داشبورد با DailyBriefing واقعی (تولید سمت کلاینت)،
سیستم هدف درختی (Goal → Project → Milestone → Week → Day → Task)، و Skill Tree
کامل با ۴ دسته و منطق لول‌آپ است. فازهای بعدی (عادت‌ها، ژورنال هوشمند، ضد
اهمال‌کاری و...) روی همین پایه اضافه می‌شوند.

**آپدیت مهم:** روش تامین `google-services.json` برای بیلد Codemagic از حالت
Environment Variable (شکننده) به حالت **فایل مستقیم داخل ریپو** تغییر کرد. اگر
قبلاً این پروژه را داشتی، فقط کافی‌ست بخش «قرار دادن google-services.json» در
پایین را دوباره بخوانی و `codemagic.yaml` جدید را جایگزین کنی.

## ⚠️ نکته مهم درباره‌ی پوشه android/

چون این پروژه بدون نصب Flutter روی سیستم من ساخته شده، پوشه‌ی استاندارد `android/`
(که معمولاً با دستور `flutter create` تولید می‌شود) در این پکیج **وجود ندارد**.
این تصمیم عمدی است: تولید `android/` توسط خود Flutter در سرور Codemagic تضمین
می‌کند که فایل‌های حساسی مثل Gradle Wrapper (که به‌صورت باینری و وابسته به نسخه‌ی
دقیق Flutter/Gradle هستند) همیشه معتبر و به‌روز باشند — دست‌ساز کردن آن‌ها بدون
اجرای واقعی Flutter می‌توانست بیلد را به‌شکلی نامعتبر و غیرقابل‌دیباگ خراب کند.
تنها بخشی که واقعاً نیاز به کنترل داشت (تنظیمات Firebase در Gradle) از طریق
پوشه‌ی `ci/` و فایل `firebase/google-services.json` مدیریت می‌شود که هر دو در
همین ریپو کامیت شده‌اند و در بیلد جایگزین می‌شوند. پس نگران نباش — نیازی به نصب
Flutter روی لپ‌تاپت نیست و بعد از اضافه‌کردن فایل واقعی فایربیس (پایین توضیح
داده شده)، هر بار که پروژه را به Codemagic بدهی، بدون هیچ تنظیم دستی اضافه‌ای
بیلد می‌شود.

## مراحل راه‌اندازی (قبل از آپلود به Codemagic)

### ۱. ساخت پروژه Firebase

۱. برو به [console.firebase.google.com](https://console.firebase.google.com) و یک پروژه جدید بساز (مثلاً `architect-app`).
۲. داخل پروژه، یک اپ Android اضافه کن با **Package name دقیقاً برابر**:
   ```
   com.farzan.architect
   ```
   (اگر می‌خواهی اسم دیگری بگذاری، هم اینجا و هم در `ci/android_app_build.gradle`
   مقدار `applicationId` و `namespace` را همزمان عوض کن — بیلد اگر این دو یکی
   نباشند، خودش با یک پیام خطای واضح متوقف می‌شود، پس اشتباه بی‌سروصدا رد نمی‌شود.)
۳. فایل `google-services.json` را دانلود کن.
۴. در Firebase Console:
   - **Authentication → Sign-in method → Email/Password** را فعال کن.
   - **Firestore Database** را بساز (Start in production mode کافی است، چون فایل
     `firestore/firestore.rules` قوانین امن را برایت نوشته).
   - محتوای `firestore/firestore.rules` را در تب **Rules** پیست و Publish کن.

### ۲. قرار دادن google-services.json (روش نهایی و پایدار)

فایل دانلودشده را **دقیقاً** جایگزین این مسیر داخل ریپو کن:
```
firebase/google-services.json
```
(الان همان‌جا یک فایل نمونه/الگو به اسم `firebase/google-services.json.example`
هست تا ساختار را ببینی — فایل واقعی‌ات باید اسمش دقیقاً `google-services.json`
باشد، بدون `.example`.)

این فایل **محرمانه نیست** (خود مستندات رسمی فایربیس هم همین را می‌گوید — امنیت
واقعی از طریق Firestore Security Rules تامین می‌شود، نه پنهان‌کردن این فایل)،
پس مشکلی ندارد که مستقیم داخل ریپو کامیتش کنی.

**چرا این روش به‌جای Environment Variable در Codemagic؟**
چون در UI فعلی Codemagic امکان آپلود فایل وجود ندارد و فقط متغیرهای متنی
(plain/base64) پشتیبانی می‌شود — که برای محتوای JSON چندخطی مستعد خطای کپی/پیست
و انکودینگ است. با کامیت‌کردن فایل مستقیم در ریپو، این لایه‌ی شکننده کاملاً حذف
می‌شود: `codemagic.yaml` فقط این فایل را از مسیر `firebase/google-services.json`
داخل همان android که در زمان بیلد ساخته می‌شود کپی می‌کند — بدون نیاز به هیچ
تنظیم اضافه در پنل Codemagic. اگر این فایل را (با محتوای واقعی) در ریپو نگذاری،
بیلد از همان مرحله‌ی اول با یک پیام فارسی واضح متوقف می‌شود، نه با خطای گنگ
Gradle در انتهای بیلد.

### ۳. آپلود در Codemagic

۱. پروژه را به‌صورت زیپ یا از طریق یک ریپوی Git (GitHub/GitLab/Bitbucket) به Codemagic بده.
۲. Codemagic باید فایل `codemagic.yaml` را خودش شناسایی کند (workflow با نام
   `architect-android`). هیچ Environment Variable Group ای لازم نیست.
۳. بیلد را اجرا کن. خروجی، فایل APK نصبی زیر مسیر
   `build/app/outputs/flutter-apk/app-release.apk` خواهد بود که Codemagic
   به‌صورت artifact قابل دانلود نمایش می‌دهد.

## ساختار داده Firestore (فاز ۱)

```
users/{uid}
  email, displayName, createdAt, xpRates

users/{uid}/daily_logs/{yyyy-MM-dd}
  studyMinutes, workoutMinutes, totalScreenTimeMinutes,
  instagramMinutes, youtubeMinutes, sleepHours, moodScore,
  xpEarned, journalWritten

users/{uid}/screen_time/{yyyy-MM-dd}
  totalScreenTimeMinutes, instagramMinutes, youtubeMinutes, lastUpdated
```

این ساختار عیناً همان چیزی است که در بخش ۱ پرامپت اصلی توضیح داده شده، تا Cloud
Function تحلیل شبانه در فاز بعد بدون تغییر ساختار داده روی همین بنا شود.

## فاز ۲ — سیستم هدف + DailyBriefing واقعی

### ⚠️ چرا بدون Cloud Functions؟
طبق بررسی دقیق، **Cloud Functions حتی در حجم مصرف رایگان هم نیاز به فعال‌سازی پلن
Blaze (پرداخت به‌ازای مصرف + کارت بانکی) دارد** — این محدودیتی از طرف گوگل است و
هیچ راه دوری ندارد. چون خواسته بودی کاملاً روی Spark Plan بمانیم، منطق
`analyzeDayAndPlanTomorrow` را **داخل خود اپ فلاتر** پیاده کردم: هر بار اپ باز
می‌شود، اگر بریفینگ امروز هنوز ساخته نشده باشد، داده‌ی دیروز را با میانگین ۷ روز
قبل‌تر مقایسه می‌کند، علت ریشه‌ای را حدس می‌زند، و متن انگیزشی می‌سازد — دقیقا
همان منطق Cloud Function توضیح‌داده‌شده در پرامپت، فقط اجراکننده‌اش کلاینت است
نه سرور. یک دکمه‌ی 🔄 «تحلیل مجدد» هم روی کارت گذاشته‌ام تا هر وقت خواستی
دستی دوباره تولیدش کنی.

### فعال‌سازی هوش مصنوعی واقعی (Gemini)
۱. به [aistudio.google.com/apikey](https://aistudio.google.com/apikey) برو و یک
   کلید رایگان Gemini بساز (کارت بانکی لازم نیست، این سرویس جدا از Firebase است).
۲. داخل اپ روی آیکون ⚙️ در بالای داشبورد بزن و کلید را وارد کن.
۳. اگر کلید را وارد نکنی، اپ همچنان کار می‌کند اما به‌جای متن تولیدشده توسط
   Gemini، یک تحلیل قالبی (بدون AI ولی مبتنی بر همان اعداد واقعی‌ات) نشان می‌دهد.

### سیستم هدف (Goal Hierarchy)
تب «اهداف» در نوار پایین اضافه شده. ساختار درختی دقیقا طبق بخش ۲ پرامپت:
هدف اصلی ← پروژه ← نقطه عطف ← هفته ← روز، و تسک در هر سطحی قابل افزودن است.
از منوی سه‌نقطه‌ی کنار هر گره می‌توانی زیرمجموعه/تسک اضافه کنی، ویرایش کنی، یا
با تایید، آن گره و تمام زیرمجموعه‌ها/تسک‌هایش را حذف کنی (حذف آبشاری).
تیک‌زدن یک تسک، XP آن را خودکار به `xpEarned` لاگ امروز اضافه می‌کند.

## فاز ۳ — XP و Skill Tree

تب سوم «مهارت‌ها» به نوار پایین اضافه شد، با ۴ زیرتب دقیقا طبق بخش ۳ پرامپت:
**دانش، بدن، ذهن، روابط**.

- هنگام ثبت‌نام، ۱۰ مهارت پیش‌فرض از لیست پرامپت خودکار ساخته می‌شود (مطالعه،
  ورزش، شطرنج، تخته‌نرد، زبان، طراحی، کتاب‌خوانی، پادکست، ژورنال‌نویسی،
  مدیتیشن) با همان نرخ‌های XP که خودت مشخص کرده بودی. دسته‌ی «روابط» عمداً
  خالی می‌ماند تا خودت مهارت‌های اجتماعی دلخواهت (مثلا «وقت با خانواده») را
  با دکمه‌ی + اضافه کنی.
- هر مهارت یک نرخ XP-به-ازای-واحد قابل‌ویرایش دارد (واحدش می‌تواند «دقیقه»،
  «بازی»، «روز» یا هرچیز دلخواه باشد) — پس هم می‌توانی مهارت‌های پیش‌فرض را
  دست بزنی، هم مهارت کاملا جدید با نرخ خودت بسازی.
- دکمه‌ی «ثبت فعالیت» روی هر کارت مهارت، مقدار فعالیت را می‌گیرد، XP را
  محاسبه می‌کند، و **در یک تراکنش امن Firestore** آن را به XP/لول مهارت اضافه
  می‌کند. وقتی XP به آستانه‌ی `۱۰۰ × لول فعلی` برسد، خودکار لول‌آپ می‌شود و یک
  اسنک‌بار جشن (🎉) نشان داده می‌شود.
- هنگام ساخت تسک در تب «اهداف»، حالا می‌توانی آن تسک را به یک مهارت وصل کنی؛
  با تیک‌زدن تسک، هم XP امروز و هم XP همان مهارت بالا می‌رود.
